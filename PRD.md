## Terminology

- **Document** — A raw source text retrieved from one of the three corpora. Length: 500–16,000 characters.
- **Excerpt** — The first 100 characters of a document; used as context for question generation only.
- **Question** — An information-seeking Hebrew question generated by an LLM based on the excerpt, classified by complexity level.
- **Answer** — A Hebrew abstractive answer generated by an LLM given the full document and question.
- **Claim** — An atomic factual statement within an answer, each accompanied by a supporting citation span from the source document.
- **Citation span** — The specific sentence or passage in the source document that supports a given claim.
- **Level** — The complexity classification of a question (0–3). See §4.2.
- **IAA** — Inter-Annotator Agreement; the proportion of items on which two independent annotators agree.
- **Pilot** — A mandatory review checkpoint after the first 50 questions are annotated.
- **Leaderboard** — MAFAT and Dicta's evaluation infrastructure that will benchmark LLM answer quality using this dataset.

---

## 1. Overview

Build a Hebrew **Abstractive QA** dataset of 1,000 question–answer pairs, sourced from three native-Hebrew corpora, intended for benchmarking LLM answer quality on the MAFAT/Dicta leaderboard.

A scoping pilot covering Hebrew Wikipedia was completed prior to this project, validating the question generation methodology and evaluation pipeline.

The work is divided into three parts:

| Part | Description | Owner |
|------|-------------|-------|
| **Part 1** | Question generation and human validation | Webiks |
| **Part 2** | Answer generation and human quality evaluation | Webiks |
| **Part 3** | Leaderboard integration | MAFAT + Dicta |

This document covers Parts 1 and 2 (Webiks scope). Part 3 is referenced for context only.

## 2. Data Sources

### 2.1 Corpora

| Corpus | Count | Format | Register |
|--------|-------|--------|----------|
| Hebrew Wikipedia | 300 documents | JSONL | Encyclopedic, factual |
| Israel HaYom | 300 documents | `.txt` (header + body) | Journalistic, current events |
| Knesset protocols | 300 documents | JSONL | Formal-spoken, sometimes political/connotative |

**Total: 900 source documents.**

### 2.2 Document Selection Criteria

- **Minimum length:** 2,000 characters
- **Maximum length:** 60,000 characters
- Selection is **random** within each source (no topical bias).
- Distribution of documents across question levels is **uniform per level** within each source.

### 2.3 Pre-existing Assets

The Wikipedia corpus (300–310 articles) is already available under `data_prep/wiki/`. Israel HaYom and Knesset protocol corpora are available under `data_prep/il-hym/` and `data_prep/knesset/` respectively.

## 3. Part 1 — Question Generation

### 3.1 Process Flow

```
900 source documents
       │
       ▼
Extract first 100 chars of each document (excerpt)
       │
       ▼
LLM (Gemini) receives excerpt → generates question + assigns level
       │
       ▼
Human annotators review: validate or correct the level classification
       │
       ▼
Questions that require reclassification → re-annotated by a second pair of annotators
       │
       ▼
1,000 validated questions with level labels
```

### 3.2 Question Generation — LLM Details

- **Methodology:** TyDi QA — Prompt-based Elicitation
- **Input to LLM:** First 100 characters of the document only (general context; not enough to answer)
- **Output from LLM:** One question + a proposed complexity level (0–3)
- **LLM:** Gemini 3
- **Purpose of excerpt-only input:** Forces the question to be genuinely information-seeking — the model cannot see enough of the document to ask a question whose answer is visible in the excerpt

### 3.3 Question Taxonomy

| Level | Type | Definition | Example |
|-------|------|------------|---------|
| **0** | Unanswerable | The question is relevant to the topic, but the document does not contain the answer. The correct model response is to declare that no answer is available. | Question about budget when only a schedule appears in the text |
| **1** | Retrieval | The answer is found explicitly in a single sentence. | "When was Committee X established?" |
| **2** | Integration / Inference | The answer requires connecting 2–3 sentences from different locations in the text. | "What arguments were raised against the law and who presented them?" |
| **3** | Abstract Synthesis | The question requires understanding the "big picture", synthesizing findings, or applying conclusions to a new scenario. | "What are the main limitations of the proposed method according to the findings?" |

### 3.4 Question Distribution Targets

| Level | Count | % of Total |
|-------|-------|-----------|
| 0 (Unanswerable) | 50 | 5% |
| 1 (Retrieval) | 50 | 5% |
| 2 (Integration) | 300 | 30% |
| 3 (Abstract) | 600 | 60% |
| **Total** | **1,000** | **100%** |

Distribution is approximately balanced across the three source corpora (≈333 questions per source).

### 3.5 Human Validation — Question Classification

- **All** 1,000 questions are reviewed by annotators.
- **Overlap:** 50% of questions are reviewed by two independent annotators (for IAA measurement).
- **IAA target:** 0.9 agreement on level classification. IAA is measured only on questions that did not require correction.
- **On disagreement:** The level is corrected and the question re-enters the annotation queue for a fresh independent annotation.
- **Deletion criterion:** Questions classified as "Not Relevant" (not connected to the document at all) are removed from the dataset.

### 3.6 Pilot Checkpoint

After the **first 50 questions** are annotated (including at least 30 Level 2/3 questions), a joint review is conducted between Webiks and MAFAT to inspect quality and refine the process before continuing.

| Field | Type | Description |
|-------|------|-------------|

| `source` | Enum | `wikipedia` / `israel_hayom` / `knesset` |
| `doc_id` | String | Unique identifier for the source document |

| `level_original` | Int | The LLM's original level classification (before correction) |

## 5. Part 2 — Answer Generation

### 5.1 Process Flow

```
1,000 validated questions (from Part 1)
       │
       ▼
LLM receives: full document + question → generates answer, claims list, citations
       │
       Note: LLM has NO context from the question-generation phase
       │
       ▼
Human annotators review: validate and correct answer, claims, citations
       │
       ▼
Save original + corrected versions
       │
       ▼
1,000 QA records with evaluated answers
```

### 5.2 Answer Generation — LLM Details

- **Input to LLM:** Full source document + validated question
- **Output from LLM:**
  1. An abstractive Hebrew answer
  2. A list of atomic claims within the answer
  3. For each claim: the supporting citation span(s) from the source document
- **LLM:** TBD / configurable
- **No shared context with Part 1:** The answer-generation model does not see the question-generation prompt or the excerpt used in Part 1.

### 5.3 Answer Quality Evaluation Dimensions

All five dimensions are scored on a 0–2 scale and applied by human annotators:

| Dimension | Score 0 | Score 1 | Score 2 |
|-----------|---------|---------|---------|
| **Coherence** | Disjointed or illogical. "Word salad", severe contradictions, or sentence order that is incomprehensible. | Understandable but choppy. Reader must work to follow the model's line of thought. | Seamless and logical. Clear beginning, middle, end. Ideas build on each other naturally. |
| **Relevance** | Answer does not address the question. | Partially relevant or vague. Touches the topic but does not directly answer. | Fully relevant and focused. Answers the question directly without unnecessary content. |
| **Completeness** | Missing critical information that the document contains. | Minor omissions. Core information present, but small supporting details are missing. | Fully complete. All key relevant information from the document is present. |
| **Faithfulness** | Hallucinations present. Contains facts invented or contradicted by the document. | Mostly faithful, but includes minor unsupported additions or personal interpretation. | Fully faithful. Every statement is supported by the document. No external knowledge. |
| **Attribution / Citations** | No citations, or citations are fake (pointing to a source that does not support the claim). | Partially attributed. Most claims are cited, but some lack support; or citations are too broad. | Fully attributed. Every claim is linked to a specific, accurate span in the source. |

**Scoring notes:**
- Penalize verbosity under Relevance: unnecessary paragraphs lower the score even if they are factually correct.
- For Level 0 (unanswerable) questions: a correct model response is "the information is not in the document." In this case, Faithfulness and Relevance are evaluated against the correctness of that declaration.
- For Attribution: a citation pointing to an entire multi-page document for a specific statistic counts as Score 1, not 2.

### 5.4 Human Evaluation — Answers

- **Coverage:** All 1,000 answers are evaluated.
- **Overlap:** 35% of answers are evaluated by two independent annotators (for IAA measurement).
- **IAA target:** 0.8 agreement.
- **Corrections:** Annotators may correct the answer text, the claims list, and/or individual citations. Both the original and corrected versions are preserved in the dataset.

## 6. Part 2 — Output Record Schema

After Part 2, each record is extended with:

| Field | Type | Description |
|-------|------|-------------|
| `answer_original` | String | Raw answer as generated by the LLM |
| `answer_corrected` | String | Annotator-corrected answer (equals `answer_original` if no correction was needed) |
| `answer_was_corrected` | Bool | Whether the annotator changed the answer |
| `claims` | Array | List of atomic claim objects (see below) |
| `coherence` | Int (0–2) | Coherence score (post-correction) |
| `relevance` | Int (0–2) | Relevance score |
| `completeness` | Int (0–2) | Completeness score |
| `faithfulness` | Int (0–2) | Faithfulness score |
| `attribution` | Int (0–2) | Attribution/citations score |

**Claim object schema:**

| Field | Type | Description |
|-------|------|-------------|
| `claim_id` | String | Unique ID within the record |
| `claim_text_original` | String | Claim as extracted by the LLM |
| `claim_text_corrected` | String | Annotator-corrected claim (equals original if unchanged) |
| `citation_span_original` | String | Citation span as provided by the LLM |
| `citation_span_corrected` | String | Annotator-corrected citation span |
| `doc_sentence_index` | Int | Index of the supporting sentence in the source document |

## 7. Label Studio Setup

### 7.1 Project Structure

Two separate Label Studio projects:

| Project | Task | Records |
|---------|------|---------|
| **Part 1 — Question Validation** | Annotators validate and correct the LLM-assigned question level | 1,000 questions |
| **Part 2 — Answer Evaluation** | Annotators score and correct LLM-generated answers + claims | 1,000 answers |

### 7.2 Part 1 Label Studio Record

```json
{
  "data": {
    "record_id": "qa_001",
    "source": "wikipedia",
    "doc_id": "article_042_12345",
    "excerpt": "<first 100 chars of document>",
    "question": "<LLM-generated Hebrew question>",
    "level_proposed": 3
  }
}
```

Labeling config captures:
- `level_validated` — 5-option radio: Level 0 / Level 1 / Level 2 / Level 3 / Not Relevant (delete)
- `level_corrected` — Bool (auto-derived: true if `level_validated ≠ level_proposed`)

### 7.3 Part 2 Label Studio Record

```json
{
  "data": {
    "record_id": "qa_001",
    "source": "wikipedia",
    "doc_text": "<full source document>",
    "question": "<validated Hebrew question>",
    "level": 3,
    "answer_original": "<LLM-generated answer>",
    "claims": [
      {
        "claim_id": "c1",
        "claim_text": "<claim>",
        "citation_span": "<supporting text>"
      }
    ]
  }
}
```

Labeling config captures:
- `answer_corrected` — free-text area pre-filled with `answer_original`
- `answer_was_corrected` — bool
- Per claim: `claim_text_corrected`, `citation_span_corrected`
- Five quality dimension scores (0/1/2 radio each)

### 7.4 Hidden Fields

The following fields are included in the data payload but **hidden from annotators** (used for traceability and QA only):
- `doc_id`, `record_id`
- `level_proposed` (Part 1 — shown only after annotator selects their own answer to avoid anchoring bias; implementation TBD)

## 8. Part 3 — Leaderboard Integration (Reference Only)

Part 3 is executed by MAFAT and Dicta. It is out of scope for Webiks. It is described here for context only.

After the dataset is complete, MAFAT and Dicta will:
1. Use the validated questions and source documents as a benchmark
2. Submit the questions to various LLMs and collect their answers
3. Evaluate model answers against the same 5 quality dimensions (Coherence, Relevance, Completeness, Faithfulness, Attribution) using automated and/or human scoring
4. Publish model rankings on a Hebrew NLP leaderboard

## 11. Dependencies and Risks

| Dependency / Risk | Impact | Mitigation |
|-------------------|--------|-----------|
| Gemini API availability and quota | Delays in question and/or answer generation | Implement retry + checkpoint logic; maintain fallback model config |
| Annotator throughput | Delays in reaching 1,000 validated pairs | Define clear daily throughput targets; build Label Studio import pipeline ahead of annotation start |
| IAA below target | Rework required on annotation batch | Monitor IAA continuously; run calibration session with annotators before full production run |
| Document quality issues (too short, garbled, off-topic) | Lower-quality questions and answers | Apply 500-char minimum filter; spot-check 5% of documents before generation |
| Level distribution skew | Dataset not meeting per-level targets | Monitor distribution during generation; adjust LLM prompts if over/under-generating specific levels |

## 13. Open Questions

1. **Level 0 question sourcing:** Level 0 questions require a document where the answer is intentionally absent. Should these be generated from truncated documents, or should the LLM be prompted specifically to produce unanswerable questions? What is the validation protocol to confirm a question is truly unanswerable?

2. **Anchoring bias in Part 1:** Should annotators see the LLM's proposed level before or after they assign their own? Hiding it until after selection reduces anchoring but complicates the Label Studio config.

3. **Part 2 model selection:** The MAFAT requirements document references "Gemini 3" for answer generation. This should be confirmed or finalized before Part 2 begins.

4. **Document overlap across questions:** Can two different questions in the dataset share the same source document, or must each document produce exactly one question?

5. **Leaderboard evaluation methodology:** Will Part 3 use the same annotators/rubric, or an automated scoring pipeline? Understanding this now ensures the dataset schema is forward-compatible.

6. **Annotation guidelines finalization:** Who owns the official annotator guidelines for question validation and answer evaluation? Webiks will draft based on this PRD; MAFAT review and sign-off required before pilot.
